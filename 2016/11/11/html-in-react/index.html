Why do I need to write HTML inside React code? Nope, you don't!

2016-11-11
ReactJs, programming

What you see within a React component is ‘technically’ not HTML.

Taking it to be HTML is the most common misconception about the React library.

Let’s take a step back. We need to first look into how the barebones of React works.

In React, we do NOT write any html directly. We use ‘React DOM’ nodes to specify how the page should be structured.

See line 5 and 6. div and h1 are actually React nodes.

For a React node, first argument are the properties and second argument is its child.

so for the ‘div’ React Node on line 5, properties are null and child is h1 React Node. h1 react node will have its own set of properties and its child.

Imagine a tree of nodes if you will.

We then take this tree of React Nodes and give it to ReactDom.render on line 10

ReactDOM knows how to convert each ‘React Node’ to its corresponding html tag. It converts each node in the tree to a html tag and attaches the resulting html structure to the id ‘app’ as specified on line 10.

That’s it.

So any html that we wish to write should be written programmatically in terms of React Nodes.

The React team decided that its painful to write ‘plain’ React (i.e using React Node’s directly).

So they decided, lets build an intermediate form of ‘html looking code’ , which can be then converted to React Nodes.

This ‘html looking code’ is called JSX or JavaScript XML, something that most people confuse for HTML.

So, the idea is that programmers write their code in JSX and a transpiler (like babel) converts it to React Nodes, which in turn gets converted to HTML by ReactDOM.

The ‘tree of nodes’ that we spoke about earlier is called a Virtual DOM.

React creates a virtual DOM using the data that you give to it. If data changes, a new virtual DOM is created. React then compares this new virtual DOM with the previous one. In case of any changes, it knows which exact nodes to update. That’s what makes React fast.
