Certifications

2017-06-11
programming, random thoughts

We all have different ways of learning.

Some people learn by coding. Some people learn by understanding the theory well and then trying parts of it. Issuing a blanket statement and calling certifications a waste of time is not right.

Here’s my take on it:

Get them and forget about them

If they help you learn something well, invest some time and get them. Certifications are a means for you to become a better programmer.

You won’t be a better programmer unless you actually code.

Don’t mention about your certifications in interviews. Don’t expect they will give you an edge while job hunting.

People hire you for what you can get done. Demonstrate that.

Adapt

2017-05-19
programming, random thoughts

As software engineers, ‘change’ is the very nature of our work.

Requirements change.

Deadlines change.

Technologies change.

Resources get re-allocated.

New versions release.

The ‘bus factor’ catches up.

The code in the new project sucks.

Essentially, shit changes.

What do you do?

Stay on your toes. Be nimble. Embrace change. Pivot.

And most importantly….ADAPT!

That’s what it comes down to.

Either adapt or get rolled over.

Jack

2017-05-17
programming, random thoughts

I do not like to classify myself as either a front-end or a back-end engineer. These terms seem mutually exclusive.

I do not like being called a full-stack engineer either. You really have to be ‘full ‘of yourself to be calling yourself that. The stack is too big for you to know it ‘fully’.

How about being a jack of a few things and being a master of few others.

Hmm. This seems to make more sense.

I want to at least be a ‘jack’ of any new technology I am learning. Irrespective of it being used on the front-end or back-end of the stack.

Any technology that I happen to work or play with, I should know enough to survive in it.

Any technology that I enjoy working on, I should constantly strive to improve in it. Eventually become a master in it.

Jack of All. Master of Some.

Best kept secrets of programmers

2016-04-23
point by point, programming

This is my collection of quotes on how to be a better programmer.

  1. Know the concepts. Solving a problem via memory or pattern recognition is much faster than solving it by reason alone.

  2. Know the tools. This is not an end in itself, but a way to maintain “flow” while programming. People who learn their tools, practice using their tools, and automate things that the tools can’t do by themselves can easily be several times as productive as those who do none of those things.

  3. Manage time. If you want to write code, write code. If you want to review a bunch of patches, review a bunch of patches. Don’t try to do all three together. Get your mind set to do one thing, then do that thing for a good block of time before you switch to doing something else.

  4. Reuse everything. Reuse ideas. Reuse code.

Point by Point - Cordova and Ionic

2016-04-14
Ionic, point by point, programming

Cordova

  1. Apache Cordova enables programmers to build applications for mobile devices using CSS3, HTML5, and JavaScript

  2. Just straight HTML,CSS and Javascript, no platform-specific ‘native’ APIs like those in Android, iOS, or Windows Phone

  3. Cordova applications execute within wrappers targeted to each platform, and rely on standards-compliant API bindings to access each device’s capabilities such as sensors, data, network status, etc.

  4. The resulting applications are hybrid, meaning that they are neither truly native mobile application (because all layout rendering is done via Web views instead of the platform’s native UI framework) nor purely Web-based (because they are not just Web apps, but are packaged as apps for distribution and have access to native device APIs).
